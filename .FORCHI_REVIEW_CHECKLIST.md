### SEC-001: Cron endpoints públicos — VERIFICAR

**Contexto:** `vercel.json` crons + `.github/workflows/cron-jobs.yml` llaman `/api/cron/*` con header `Authorization: Bearer ${CRON_SECRET}`.

**Riesgo:** CRON_SECRET debe estar configurado como "Secret" en Vercel + GitHub. Si no lo está, cualquier usuario puede:
- POST a `/api/cron/status`
- POST a `/api/cron/ingest`
- POST a `/api/cron/recalculate`
- POST a `/api/cron/simulate`

Esto podría:
- Inundar bases de datos con cálculos innecesarios
- Disparar requests a APIs externas (football-data.org, API-Football)

**Action item:** 
- [ ] Verificar Vercel → Settings → Environment Variables que `CRON_SECRET` está bien configurado
- [ ] Verificar GitHub → Settings → Secrets que `CRON_SECRET` está bien configurado (mismo valor)
- [ ] Verificar que `app/api/cron/*/route.ts` valida CRON_SECRET en cada handler

## 🟡 Warnings — Performance / Reliability

### PERF-001: Rate limits en producción

**Contexto:** football-data.org free tier = 10 req/min. Si las crons hacen 4+ requests cada una + el dashboard hace más, podemos exceder el rate limit.

**Action item:**

### PERF-002: Cold starts de Vercel functions

**Contexto:** `maxDuration: 30-60s` para varias rutas. Si las funciones se duermen (Vercel cold starts), el primer request puede tardar 5-10s.

**Action item:**
- [ ] Verificar si Vercel ha escalado a Pro para evitar cold starts
- [ ] Considerar warmup ping a crons para mantener functions warm

### PERF-003: Bundle size y Core Web Vitals

**Contexto:** Stack incluye Tailwind + framer-motion + lucide-react + react + next. Bundle puede ser grande.

**Action item:**
- [ ] Auditar bundle size via `next build --turbo`
- [ ] Lighthouse en producción para LCP/CLS/INP
- [ ] Considerar lazy-load de framer-motion en mobile

### PERF-004: Storage strategy

**Contexto:** README dice "almacenamiento en memoria con persistencia en archivo". Esto significa memoria volatil entre deploys de Vercel.

**Riesgo:** Cada deploy pierde estado. Cron jobs necesitan re-ejecutar para re-poblar.

**Action item:**
- [ ] Considerar migrar a Vercel Postgres / Vercel KV para persistencia
- [ ] O documentar que el cache file está en Vercel persistent filesystem (si aplica)
